# プロジェクトファイル構成ガイド

## 1. はじめに

このドキュメントは、`Market_News_Project` のファイルとディレクトリ構造を理解するためのガイドです。プロジェクトには多くのファイルが含まれていますが、その役割によって「開発で重要なファイル」と「原則として直接触らないファイル」に大別できます。

このガイドを参考に、どこに何があるかを把握し、効率的に開発を進めてください。

---

## 2. ファイル分類の概要

| カテゴリ | ディレクトリ / ファイル例 | 重要度 | 説明 |
|:---|:---|:---:|:---|
| **ソースコード** | `src/` | ★★★★★ | アプリケーションのコアロジック。機能追加や修正はここが中心。 |
| **スクリプト** | `scripts/` | ★★★★☆ | 開発支援、データ移行、診断などの補助的なスクリプト群。 |
| **テスト** | `tests/` | ★★★★☆ | ユニットテストや統合テスト。コードの品質を保証する。 |
| **設定ファイル** | `requirements.txt`, `pyproject.toml`, `.flake8` | ★★★★★ | プロジェクトの依存関係や静的解析、フォーマットのルールを定義。 |
| **CI/CD** | `.github/workflows/` | ★★★☆☆ | GitHub Actionsによる自動テストやデプロイの定義。 |
| **ドキュメント** | `docs/`, `README.md` | ★★★☆☆ | プロジェクトの仕様、設計、運用マニュアルなど。 |

| **生成物** | `build/`, `public/`, `data/` | ★☆☆☆☆ | プログラム実行によって自動生成されるファイル。**原則として直接編集しない。** |

## 3. 主要なファイルとディレクトリの詳細

### 【最重要】 `src/` - ソースコード

アプリケーションの心臓部です。Pythonコードのほとんどがここに配置されています。

- **`src/core/`**: 中核機能
  - `news_processor.py`: ニュース記事の収集、AI要約、出力までの一連の処理フローを実装したメインロジック。
  - `social_content_generator.py`: SNS投稿用のコンテンツを生成するロジック。
- **`src/database/`**: データベース関連
  - `database_manager.py`: データベースとの接続やCRUD操作を管理。
  - `models.py`: データベースのテーブル定義。
- **`src/podcast/`**: ポッドキャスト機能
  - `podcast_processor.py`: ポッドキャスト全体の処理を管理。
  - `script_generation/`: AIによる台本生成ロジック。
  - `audio/`: 音声合成や音声ファイルの加工処理。
  - `publisher/`: RSSフィードの生成や外部サービスへの配信処理。
- **`src/html/`**: Webページ生成
  - `html_generator.py`: 収集・要約したニュースからHTMLファイルを生成。
- **`src/config/`**: アプリケーション設定
  - `app_config.py`: 環境変数から設定を読み込み、アプリケーション全体で利用可能にするクラス。
- **`src/error_handling/`**: エラー処理
  - `custom_exceptions.py`: プロジェクト固有のカスタム例外を定義。

### `scripts/` - スクリプト群

開発や運用を補助するためのコマンドラインツール。

- **`scripts/core/`**: メイン処理の実行スクリプト。
- **`scripts/dev/`**: 開発中に便利なテスト用スクリプト。
- **`scripts/diagnostics/`**: GCPの認証情報やAPI設定などを診断するツール。問題発生時の切り分けに役立つ。
- **`scripts/migrations/`**: データベースのスキーマ変更などを行うマイグレーションスクリプト。

### `tests/` - テストコード

コードの品質と動作を保証するためのテスト。

- **`tests/unit/`**: 各モジュール（関数やクラス）が単体で正しく動作するかを検証するユニットテスト。
- **`tests/integration/`**: 複数のモジュールを連携させた場合の動作を検証する統合テスト。

### (ルートディレクトリ) - 設定ファイル

- **`requirements.txt`**: プロジェクトが依存するPythonライブラリの一覧。`pip install -r requirements.txt`で環境を構築する。
- **`pyproject.toml`**: `pytest`などのツール設定を記述する、比較的新しい形式の設定ファイル。
- **`.flake8`, `.coveragerc`**: コードのスタイルチェックやテストカバレッジ測定ツールの設定ファイル。
- **`README.md`**: プロジェクトの概要、セットアップ方法、使い方を記した最重要ドキュメント。

### `.github/` - CI/CD

GitHub Actionsを利用した自動化ワークフローの定義ファイル（YAML）。

- `main.yml`: `main`ブランチにコードがプッシュされた際に、テストや本番環境へのデプロイを実行する。
- `social-content-preview.yml`: ソーシャルメディア投稿のプレビューを生成するなど、特定のタスクを実行する。

### `docs/` - ドキュメント

- プロジェクトの要件定義、API仕様、運用マニュアル、調査レポートなどが格納されている。新しい機能の背景や設計思想を理解するのに役立つ。

## 4. 原則として直接編集しないファイル

以下のディレクトリに含まれるファイルは、プログラムによって自動的に生成されたり、過去の遺物であったりするため、開発者が手動で編集・管理する必要は基本的にありません。

### `build/`, `public/`, `data/` - 生成物

- **`build/`**: ビルドプロセスの中間生成物。
- **`public/`**: GitHub Pagesで公開されるHTMLファイルや、その他のプレビュー用ファイルが格納される。`src/html/html_generator.py`などのプログラムによって生成される。
- **`data/`**: 処理済みの記事データ（JSON）などが保存される。

**なぜ編集しないのか？**
これらのファイルはソースコード（`src`など）から自動生成されるため、直接編集しても次回のプログラム実行時に上書きされてしまいます。変更を加えたい場合は、生成元となっているソースコードやテンプレートを修正する必要があります。

### `logs/` - ログファイル

- アプリケーション実行時の動作ログが保存されます。エラーの原因調査や、意図した通りに処理が実行されたかを確認するために使います。

### `Archive/` - アーカイブ

- **役割**: 過去に作成されたが、現在は使われていないドキュメント、コード、レポートなどが保管されています。
- **取り扱い**: プロジェクトの歴史的経緯を知る上で参考になることはあるかもしれませんが、現在の開発や運用には全く影響しません。**基本的に無視して問題ありません。**

---

以上がプロジェクトの主要なファイル構成です。新しい機能を追加・修正する際は、まず`src/`ディレクトリの関連するモジュールを探し、必要に応じて`tests/`にテストを追加する、という流れが基本になります。
